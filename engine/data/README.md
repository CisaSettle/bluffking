# Engine static data

## `preflop_v2.json` (current — CFR approx.-equilibrium chart)

6-max, 100bb preflop ranges per ADR-043 §3.2. **Clean-room, no third-party
data.** The previous edition of this README asserted a "GTO Wizard public charts
+ PokerCoaching" provenance — that was **stale and wrong**: it re-introduced the
exact licensing taint the clean-room generator was built to remove, and it
contradicted the file's own `_source` field. This README now matches the data.

- **Source / method:** BluffKing **clean-room** generated by
  `engine/examples/gen_preflop_ranges.rs`, which runs a genuine
  **discounted-regret CFR variant (DCFR-style coefficients) approximate-equilibrium
  solve** over a SIMPLIFIED 6-max/100bb preflop tree
  (`engine/src/solver/preflop_cfr.rs`). The update form is a discounted-regret
  *variant* using the canonical DCFR `(α,β,γ) = (1.5, 0, 2)` coefficients, **not**
  the canonical DCFR update (see `preflop_cfr.rs` for the exact form). Terminal
  EVs use BluffKing's OWN
  Monte-Carlo equity engine (`engine::solver::equity`) for all-in equity, plus a
  documented bounded **equity-realization factor** `R ∈ [0.70, 1.12]` for
  see-flop terminals. **No third-party charts, solver outputs, or commercial
  range datasets were used or consulted.**
- **Honest label:** **"CFR approx. equilibrium · equity-realization"** — this is
  genuine equilibrium math, but it is an equilibrium of a *simplified model* (the
  engine has no postflop solver), **NOT a true postflop GTO solve**. It is NEVER
  labelled "GTO" anywhere in code, badge, or copy. Honest fidelity ceiling:
  *directionally sound, model-dependent.*
- **Schema:** the in-file `version` field is now **3** (mixed frequencies). The
  filename stays `preflop_v2.json` (the consumer's `include_str!` path), but
  `engine::solver::PREFLOP_V` (and the paired `server::coach::PROMPT_V`) are now
  **3** to invalidate the `coach_hints` cache per ADR-043 §5 (see "Versioning
  note" below). Each cell is hero's CONTINUE frequency in `[0,1]` (mixed
  strategies — an upgrade over the v1/v2 binary `{0.0, 1.0}` chart). Absent = fold.
- **Shape:** 6 positions (UTG, MP, CO, BTN, SB, BB) × **5** action buckets (RFI,
  facing_open, vs_3bet, vs_4bet, facing_limp). BB has no RFI (cannot open
  unopened); UTG has no facing_open (acts first) — empty objects are intentional.

### All-in equity matrix — fixed representative combo (honesty note, codex F3)

The terminal all-in equities come from a deterministic 169×169 matrix built by
`preflop_cfr::EquityMatrix::build`. HONESTY: each cell is the equity of ONE fixed
representative combo per class (hero always Spades/Hearts, villain always
Diamonds/Clubs), NOT the average over all legal combo pairs for the class pair.
Suits are irrelevant for vs-random equity, but for a specific class-vs-class
showdown a SHARED suit slightly lowers both sides' flush outs — and the fixed
hero/villain suit split means a same-suit clash is never sampled. The bias is
sub-1% and `combined_pct` is quantized to whole percent (chart frequencies then
rounded to 2 dp), so the published ranges are unaffected; it is honestly a
"fixed-representative-combo" matrix, not an exact class-averaged one.

### Model parameters (FIXED inputs, NOT solved — so the output is not over-claimed)

The bet-sizing tree and the realization model are *parameters*, documented in the
generator + the JSON `_source`: open 2.5bb, 3-bet 3.2× open, 4-bet 2.2× 3-bet,
5-bet jam; villain 3-bet freq ~11%, 4-bet freq ~5.5%; a best-of-N-defenders
position model + a minimum-defense-frequency floor; `R ∈ [0.70, 1.12]`. A
different sizing/`R` model yields different ranges.

### Poker-sanity invariants (enforced + guarded)

`engine/tests/preflop_chart_contract.rs` is a RED-first guard: AA/KK/AKs in every
RFI, 72o nowhere, `vs_4bet ⊆ vs_3bet` per position, RFI widths widen
UTG ⊆ MP ⊆ CO ⊆ BTN, AA in every non-empty bucket, and a clean-room/honest-label
attestation (no third-party token; names the CFR + equity-realization method;
never positively claims "GTO").

### Reproducibility

Deterministic: a fixed equity-matrix seed + fixed MC trial count + fixed DCFR
iteration count + deterministic traversal (canonical 169 order, no `HashMap`
iteration) + a stable hand-written JSON emitter ⇒ **byte-identical** regen, in
both debug and release.

```bash
# Regenerate (release is ~10× faster; output is identical in debug):
cargo run -p engine --example gen_preflop_ranges --release > engine/data/preflop_v2.json
# Verify reproducibility (no diff against the committed file):
cargo run -p engine --example gen_preflop_ranges --release | diff - engine/data/preflop_v2.json
```

### Versioning note

ADR-043 §3.2 + §5 require a `PREFLOP_V` + `PROMPT_V` bump whenever the data
content changes (to invalidate the `coach_hints` cache). The DCFR commit
regenerated `preflop_v2.json` IN PLACE and bumped only the in-file `version` field
2 → 3, deferring the constant bump to this review.

**Resolved (2026-06-25, codex F4): `engine::solver::PREFLOP_V` and
`server::coach::PROMPT_V` are now both `3`** (model column `local-solver-v3`). The
chart content changed twice — the DCFR mixed-frequency rewrite AND the F2
all-in-symmetry math fix that shifted the `vs_4bet` / `vs_3bet` ranges — so per
ADR-043 §5 the cache MUST be invalidated; serving stale `prompt_v=2` hints over
the new chart would mis-grade hands. `server/src/main.rs` asserts
`PROMPT_V == PREFLOP_V` at startup, so the two move in lock step. The filename
stays `preflop_v2.json` (the `include_str!` path is not a cache key).
*Architect / cross-vendor sign-off requested for the prod cache impact* — see the
PR/handoff notes; the change is conservative (old cache rows are ignored, not
deleted, and re-derived on next coach query).
